Ergonominador – IoT Ergonomic Monitoring System

Full documentation: https://ergonominador.vercel.app/pages/documentation/

View Live Demo →

Overview

Django system integrating IoT sensors via MQTT to monitor ergonomic workspace conditions. Architecture follows: Hardware → MQTT Broker → Django Backend → Frontend Dashboard.

Authors

  • Leonardo Guevara
  • Phol Castañeda

Tech Stack

  • Backend: Django 5.1.2 + SQLite
  • IoT Communication: MQTT (Paho-MQTT + HiveMQ Cloud)
  • Frontend: jQuery + Chart.js + Bootstrap 4
  • Hardware: ESP32 + MicroPython
  • Sensors: LM35DZ (temperature), HC-SR04 (ultrasound), LDR (light)
  • Protocol: MQTT over TLS (port 8883)

Key Components

MQTT Client

Permanent subscription to topics, sensor data reception, and database persistence. Auto-starts in Django deployment via daemon thread.

Subscribed Topics:

  • sensorsPHOLLEO/{temp|sonido|luz} - Sensor readings
  • alertPHOLLEO/{distancia|temp|luz} - Hardware-generated alerts
  • alertPHOLLEO/postura/{Verde|Amarillo|Rojo} - Posture traffic light states

Data Models

  • SensorTemp/SensorSonido/SensorLuz: Store historical readings (value + timestamp)
  • Alert: Notifications classified by type with seen flag
  • Postura: Records time in each traffic light state

REST API

  • /get-alerts/ - Returns latest unseen alert by type + current traffic light state
  • /get-sensor-data/ - Last 5 minutes data + time aggregates by traffic light color
  • /sensors_view/ - Real-time dashboard with Chart.js graphs (5s polling)

IoT Flow

Hardware → Backend:

  1. Physical sensors (temperature, ultrasound, photoresistor) connected to microcontroller
  2. Microcontroller publishes data via WiFi to public MQTT broker (HiveMQ)
  3. Django backend receives messages in real-time and stores in SQLite
  4. Frontend queries API via polling and visualizes historical data

Traffic Light System (2 LEDs):

Hardware measures screen distance with ultrasonic sensor. Based on time in adequate/inadequate posture, activates indicator LEDs:

  • Green LED: Correct posture (>50cm distance)
  • Yellow LED: Preventive warning (40-50cm or approaching time limit)
  • Red LED: Incorrect posture (<40cm)

Dashboard shows time proportion in each state via donut chart for posture habit analysis.

Main Endpoints

  • http://localhost:8000/ - Main page
  • http://localhost:8000/sensors_view/ - Dashboard with graphs
  • http://localhost:8000/pages/documentation/ - Technical documentation
  • http://localhost:8000/get-alerts/ - Alerts API
  • http://localhost:8000/get-sensor-data/ - Sensor data API

FocusHub – Productivity Management Platform

Link to the Repository: https://github.com/LeonardoG2005/FocusHub

View Live Demo →

Overview

FocusHub is a productivity monorepo combining an Angular 19 frontend and a NestJS 11 backend. It covers task management, calendar planning, concentration techniques, focus sessions, productivity analytics, and JWT authentication in a single integrated platform.

Authors

  • Leonardo Guevara

Tech Stack

  • Frontend: Angular 19 (standalone components + signals)
  • Backend: NestJS 11 + TypeORM
  • Database (main branch): SQLite local database (focushub.db)
  • Authentication: JWT with guard-protected routes and endpoints
  • Optional Orchestration: Docker Compose with frontend, backend, MySQL, and ELK
  • Observability: ELK stack configuration (Elasticsearch, Logstash, Kibana)

Key Components

Frontend Modules (Angular)

  • Routes: /, /sign-up, /log-in, /tasks, /calendar, /techniques, /stats
  • AuthService + TokenService: Registration/login/logout and JWT lifecycle in localStorage
  • TaskService + EventService: CRUD operations with reactive UI state
  • TechniqueService: Focus timer, active session recovery, and task association per focus session
  • StatsService: Productivity statistics consumption for the analytics views

Backend Modules (NestJS)

  • auth, users, tasks, events, categories, productivity, ambient-sound, reminders
  • Swagger Docs: API documentation exposed at /api
  • Persistence: TypeORM configured for SQLite in the main branch

Repository Structure

  • FocusHub-app/ - Angular frontend
  • focus-hub-backend/ - NestJS backend
  • elk/ - Logstash/ELK configuration
  • docker-compose.yml - Multi-service orchestration

Architecture Flow

User Workflow:

  1. Users register or log in and receive JWT credentials
  2. Frontend sends authenticated requests to protected API modules
  3. Backend modules process tasks, events, categories, and productivity entities
  4. TypeORM persists data in SQLite (main setup) and serves analytics endpoints
  5. Statistics and active focus session state feed the productivity dashboards

Execution Modes:

  • Local: Backend on http://localhost:3000 and frontend on http://localhost:4200
  • Containerized: Docker Compose can run frontend, backend, MySQL, and ELK services together

Main Endpoints

  • POST /auth/register and POST /auth/login - Authentication
  • GET /users/me - Current authenticated user
  • POST/GET/PATCH/DELETE /tasks - Task management
  • POST/GET/PUT/DELETE /events and GET /events/by-date - Calendar events
  • GET /productivity/stats - Productivity analytics
  • GET /api - Swagger API docs
